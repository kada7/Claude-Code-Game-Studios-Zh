# Agent 测试规范：engine-programmer

## Agent 摘要
领域：渲染管线、物理集成、内存管理、资源加载和核心引擎框架。
不负责：游戏机制（gameplay-programmer）、编辑器/调试工具 UI（tools-programmer）。
模型层级：Sonnet（默认）。
未分配门控 ID。

---

## 静态断言（结构性）

- [ ] `description:` 字段存在且具有领域特定性（引用渲染/内存/引擎核心）
- [ ] `allowed-tools:` 列表包含 Read、Write、Edit、Bash、Glob、Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] Agent 定义不声称对游戏机制或工具 UI 拥有权限

## 测试用例

### 用例 1：领域内请求 — 适当的输出
**输入：** "为弹丸实现自定义对象池以避免每帧分配。"
**预期行为：**
- 生成引擎级对象池实现，包含获取/释放接口
- 池针对弹丸对象类型进行类型化，使用预分配的固定大小存储
- 提供线程安全说明（或明确标记为仅单线程并给出理由）
- 根据编码标准包含公共 API 的文档注释
- 输出与项目配置的引擎和语言兼容

### 用例 2：领域外请求 — 正确重定向
**输入：** "添加一个暂停菜单屏幕，包含音量滑块和'返回主菜单'按钮。"
**预期行为：**
- 不生成 UI 屏幕代码
- 明确说明菜单屏幕属于 `ui-programmer`
- 将请求重定向到 `ui-programmer`
- 可以注明它可以为 ui-programmer 提供引擎级音频音量 API 端点以供调用

### 用例 3：内存泄漏诊断
**输入：** "每次关卡加载内存使用量增加约 50MB 且从不释放。我们怀疑是资源加载系统的问题。"
**预期行为：**
- 生成系统化的诊断方法：引用计数审计、资源句柄生命周期检查、缓存失效审查
- 识别可能的原因（孤立的资源句柄、循环引用、从不驱逐的缓存）
- 为识别的泄漏模式生成具体修复方案
- 提供验证修复的测试（加载前的内存基线，卸载后测量，确认返回基线）

### 用例 4：跨领域协调 — 共享系统优化
**输入：** "我需要优化物理粗检测阶段，但游戏系统与物理查询 API 紧密耦合。"
**预期行为：**
- 不单方面更改物理查询 API 表面（这会破坏 gameplay-programmer 的代码）
- 与 `lead-programmer` 协调以安全地规划变更
- 提出迁移路径：新的优化 API 与旧 API 并存，包含弃用期
- 在继续之前记录协调要求

### 用例 5：上下文传递 — 检查引擎版本引用
**输入：** 上下文中提供引擎版本引用（Godot 4.6）。请求："为项目设置默认物理引擎。"
**预期行为：**
- 读取引擎版本引用并注意 Godot 4.6 变更：Jolt 物理现在成为默认
- 生成考虑 Jolt 作为默认变更的配置指导（4.6 迁移说明）
- 标记 GodotPhysics 和 Jolt 之间可能影响现有代码的任何 API 差异
- 不推荐已弃用或 4.6 之前的物理设置步骤，除非注明它们适用于旧版本

## 协议合规性

- [ ] 保持在声明的领域内（渲染、物理、内存、资源加载、核心框架）
- [ ] 将 UI/菜单请求重定向到 ui-programmer
- [ ] 返回结构化发现（实现代码、诊断步骤、迁移计划）
- [ ] 在更改共享 API 表面之前与 lead-programmer 协调
- [ ] 在建议引擎特定 API 之前检查引擎版本引用
- [ ] 为修复提供测试证据（修复前后的内存、性能测量）

## 覆盖范围说明
- 对象池（用例 1）必须在 `tests/unit/engine/` 中包含单元测试
- 内存泄漏诊断（用例 3）应在 `production/qa/evidence/` 中生成证据工件
- 引擎版本检查（用例 5）确认 Agent 将 VERSION.md 视为权威来源，而非 LLM 训练数据
