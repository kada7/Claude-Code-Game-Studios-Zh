# 技能测试规范：/skill-improve

## 技能摘要

`/skill-improve` 在技能文件上运行自动化的测试-修复-重新测试改进循环。它调用 `/skill-test static`（可选择性地调用 `/skill-test category`）建立基线分数，诊断失败的检查，针对 SKILL.md 文件提出具体修复建议，询问“我可以将改进写入 [技能路径] 吗？”，应用修复，并重新运行测试以确认改进。

如果提出的修复使技能变得更糟（回归），则恢复修复（经用户确认）而非应用。如果技能已完美（0 个失败），则技能立即退出而不做更改。无导演关卡适用。裁决结果：IMPROVED（分数上升）、NO CHANGE（无改进可能或用户拒绝）或 REVERTED（修复已应用但导致回归并已恢复）。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证 — 无需测试固件。

- [ ] 具有必需的前置元数据字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 具有 ≥2 个阶段标题
- [ ] 包含裁决关键词：IMPROVED、NO CHANGE、REVERTED
- [ ] 包含在应用修复前的“我可以写入吗”协作协议语言
- [ ] 具有下一步交接（例如，运行 `/skill-test spec` 以验证行为合规性）

---

## 导演关卡检查

无。`/skill-improve` 是元实用技能。无导演关卡适用。

---

## 测试用例

### 用例 1：理想路径 — 技能有 2 个静态失败，都被修复，IMPROVED

**测试固件：**
- `.claude/skills/some-skill/SKILL.md` 有 2 个静态失败：
  - 检查 4：没有“我可以写入吗”语言，尽管 allowed-tools 中包含 Write
  - 检查 5：末尾没有下一步交接

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 技能运行 `/skill-test static some-skill` — 基线：7 项检查中 5 项通过
2. 技能诊断 2 个失败检查（4 和 5）
3. 技能提出修复建议：
   - 在适当阶段添加“我可以写入吗”语言
   - 在末尾添加下一步交接部分
4. 技能询问“我可以将改进写入 `.claude/skills/some-skill/SKILL.md` 吗？”
5. 应用修复；重新运行 `/skill-test static some-skill` — 现在 7/7 检查通过
6. 裁决为 IMPROVED (5→7)

**断言：**
- [ ] 在进行任何更改前建立基线分数 (5/7)
- [ ] 诊断两个失败检查并在提议的修复中解决
- [ ] 在应用修复前询问“我可以写入吗”
- [ ] 重新测试确认改进 (7/7)
- [ ] 裁决为 IMPROVED，并显示改进前后的分数

---

### 用例 2：修复导致回归 — 分数对比显示回归，REVERTED

**测试固件：**
- `.claude/skills/some-skill/SKILL.md` 有 1 个静态失败（缺少交接）
- 提议的修复意外移除了裁决关键词部分（引入新的失败）

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 基线：7 项检查中 6 项通过（1 个失败：缺少交接）
2. 技能提出修复并询问“我可以写入改进吗？”
3. 应用修复；重新运行测试
4. 重新测试结果：7 项检查中 5 项通过（修复了交接但破坏了裁决关键词）
5. 技能检测到回归：分数下降
6. 技能询问用户：“修复导致回归 (6→5)。我可以恢复更改吗？”
7. 用户确认；恢复更改；裁决为 REVERTED

**断言：**
- [ ] 在最终确定前将重新测试分数与基线进行比较
- [ ] 当分数下降时检测到回归
- [ ] 要求用户确认恢复（非自动）
- [ ] 用户确认后恢复文件
- [ ] 裁决为 REVERTED

---

### 用例 3：具有分类分配的技能 — 基线捕获两个分数

**测试固件：**
- `.claude/skills/gate-check/SKILL.md` 是一个关卡技能，有 1 个静态失败和 2 个分类（G 准则）失败
- `tests/skills/quality-rubric.md` 具有“关卡技能”部分

**输入：** `/skill-improve gate-check`

**预期行为：**
1. 技能运行静态和分类测试建立基线：
   - 静态：7 项检查中 6 项通过
   - 分类：5 个 G 准则中 3 个通过
2. 合并基线：12 项中 9 项通过
3. 技能诊断所有 3 个失败并提出修复
4. “我可以将改进写入 `.claude/skills/gate-check/SKILL.md` 吗？”
5. 应用修复；重新运行两种测试类型
6. 重新测试：静态 7/7，分类 5/5 = 12/12
7. 裁决为 IMPROVED (9→12)

**断言：**
- [ ] 在基线中捕获静态和分类分数
- [ ] 使用合并分数进行比较（不仅仅是单一类型）
- [ ] 在提议的修复中解决所有 3 个失败
- [ ] 重新测试确认两种分数类型的改进
- [ ] 裁决为 IMPROVED，显示合并前后的分数

---

### 用例 4：技能已完美 — 无需改进

**测试固件：**
- `.claude/skills/brainstorm/SKILL.md` 没有静态失败
- 分类分数也是 5/5（如果适用）

**输入：** `/skill-improve brainstorm`

**预期行为：**
1. 技能运行 `/skill-test static brainstorm` — 7/7 检查通过
2. 如果适用分类：5/5 准则通过
3. 技能输出：“无需改进 — brainstorm 完全合规”
4. 技能退出而不提出任何更改
5. 不询问“我可以写入吗”；不修改任何文件
6. 裁决为 NO CHANGE

**断言：**
- [ ] 确认 0 个失败后技能立即退出
- [ ] 显示“无需改进”消息
- [ ] 不提出任何更改
- [ ] 不询问“我可以写入吗”
- [ ] 裁决为 NO CHANGE

---

### 用例 5：导演关卡检查 — 无导演关卡；skill-improve 是元实用技能

**测试固件：**
- 技能至少有 1 个静态失败

**输入：** `/skill-improve some-skill`

**预期行为：**
1. 技能运行测试-修复-重新测试循环
2. 不产生导演 Agent
3. 输出中不出现关卡 ID

**断言：**
- [ ] 不调用导演关卡
- [ ] 不出现关卡跳过消息
- [ ] 裁决为 IMPROVED、NO CHANGE 或 REVERTED — 无关卡裁决

---

## 协议合规性

- [ ] 在提出任何更改前始终建立基线分数
- [ ] 在输出中显示改进前后的分数对比
- [ ] 在应用任何修复前询问“我可以写入吗”
- [ ] 通过比较重新测试分数与基线检测回归
- [ ] 在恢复前要求用户确认（非自动）
- [ ] 以 IMPROVED、NO CHANGE 或 REVERTED 裁决结束

---

## 覆盖率说明

- 改进循环设计为每次调用仅运行一个修复-重新测试周期；运行多个迭代需要重新调用 `/skill-improve`。
- 行为合规性（规范模式测试结果）不包含在改进循环中 — 仅自动化结构（静态）和分类分数。
- 无法读取技能文件（权限错误或文件缺失）的情况未测试；这将在建立基线前导致错误。
