## 阶段 1：解析参数

- `version`: 要生成说明的发布版本（例如，`1.2.0`）
- `--style`: 输出风格 — `brief`（要点）、`detailed`（带上下文）、`full`（带开发者评论）。默认：`detailed`。

如果未提供版本，在继续之前询问用户。

---

## 阶段 2：收集变更数据

- 如果存在，读取 `production/releases/[version]/changelog.md` 处的内部变更日志
- 同时检查 `docs/CHANGELOG.md` 获取相关版本条目
- 作为备选，运行 `git log` 获取上一个发布标签和当前标签/HEAD 之间的记录
- 读取 `production/sprints/` 中的冲刺回顾以获取上下文
- 读取 `design/balance/` 中的任何平衡性变更文档
- 从 QA 读取 bug 修复记录（如果可用）

**如果没有可用的变更日志数据**（`production/releases/[version]/changelog.md` 和 `docs/CHANGELOG.md` 中都没有此版本的条目，且 git log 为空或不可用）：

> "未找到 [version] 的变更日志数据。先运行 `/changelog [version]` 生成内部变更日志，然后重新运行 `/patch-notes [version]`。"

裁决：**阻塞** — 在此停止，不生成说明。

---

## 阶段 2b：检测语调和模板

**语调指南检测** — 在起草说明之前，检查写作风格指南：

1. 检查 `.claude/docs/technical-preferences.md` 中是否有任何 "tone"、"voice" 或 "style" 字段或章节。
2. 检查 `docs/PATCH-NOTES-STYLE.md` 是否存在。
3. 检查 `design/community/tone-guide.md` 是否存在。
4. 如果任何来源包含语调/声音/风格说明，提取它们并应用到生成说明的语言和框架中。
5. 如果到处都找不到语调指南，默认使用：
   玩家友好、非技术语言；热情但不夸张；
   关注玩家体验到的内容，而不是开发者更改的内容。

**模板检测** — 检查是否存在补丁说明模板：

1. 全局搜索 `docs/patch-notes-template.md` 和 `.claude/docs/templates/patch-notes-template.md`。
2. 如果在任一位置找到，读取它并将其用作阶段 4 的输出结构
   而不是内置的风格模板（Brief / Detailed / Full）。用分类数据填充模板的各个部分。
3. 如果未找到，使用阶段 4 中定义的内置风格模板。

---

## 阶段 3：分类和转换

将所有变更分类到面向玩家的类别中：

- **新内容**：新功能、地图、角色、物品、模式
- **游戏性变更**：平衡调整、机制变更、进度变更
- **生活质量**：UI 改进、便利功能、无障碍功能
- **Bug 修复**：按系统分组（战斗、UI、网络等）
- **性能**：玩家可能注意到的优化改进
- **已知问题**：关于未解决问题的透明度

将开发者语言转换为玩家语言：

- "重构伤害计算管道" → "改进命中检测准确性"
- "修复库存管理器中的空引用" → "修复打开库存时的崩溃"
- "减少战斗循环中的 GC 分配" → "提升战斗性能"
- 移除对玩家没有影响的纯粹内部变更
- 保留平衡变更的具体数值（伤害：50 → 45）

---

## 阶段 4：生成补丁说明

### Brief 风格
```markdown
# 补丁 [版本] — [标题]

**新增**
- [功能 1]
- [功能 2]

**变更**
- [平衡/机制变更，包含变更前 → 变更后的数值]

**修复**
- [Bug 修复 1]
- [Bug 修复 2]

**已知问题**
- [问题 1]
```

### Detailed 风格
```markdown
# 补丁 [版本] — [标题]
*[日期]*

## 亮点
[1-2 句总结最令人兴奋的变更]

## 新内容
### [功能名称]
[2-3 句描述该功能以及为什么玩家应该感到兴奋]

## 游戏性变更
### 平衡性
| 变更 | 变更前 | 变更后 | 原因 |
| ---- | ---- | ---- | ---- |
| [物品/技能] | [旧数值] | [新数值] | [简要理由] |

### 机制
- **[变更]**：[变更内容和原因的说明]

## 生活质量
- [带上下文的改进]

## Bug 修复
### 战斗
- 修复了[玩家遇到的情况的描述]

### UI
- 修复了[描述]

### 网络
- 修复了[描述]

## 性能
- [玩家会注意到的改进]

## 已知问题
- [问题及可用的变通方法]
```

### Full 风格
包含 Detailed 的所有内容，以及：
```markdown
## 开发者评论
### [主题]
> [对重大变更的开发者见解 — 为什么做出此变更、考虑过什么、
> 团队学到了什么。用第一人称团队声音撰写。]
```

---

## 阶段 5：审查输出

检查生成的说明：

- 没有内部术语（用玩家友好的语言替换技术术语）
- 没有引用内部系统、工单或冲刺编号
- 平衡变更包含变更前/变更后的数值
- Bug 修复描述玩家体验，而不是技术原因
- 语调与游戏的声音匹配（根据游戏风格调整正式程度）

---

## 阶段 6：保存补丁说明

向用户展示完整的补丁说明，以及：按类别统计的变更数量，以及任何被排除的内部变更（供审查）。

询问："我可以将这些补丁说明写入 `docs/patch-notes/[version].md` 吗？"

如果同意，将文件写入 `docs/patch-notes/[version].md`，如果需要则创建目录。同时写入 `production/releases/[version]/patch-notes.md` 作为内部存档副本。

---

## 阶段 7：后续步骤

裁决：**完成** — 补丁说明已生成并保存。

- 在发布之前运行 `/release-checklist` 以验证所有其他发布门槛是否满足。
- 公开发布之前与 community-manager 分享补丁说明草案进行语调审查。
