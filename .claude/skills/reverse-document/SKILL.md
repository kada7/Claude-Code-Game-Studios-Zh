# 反向文档化

此 skill 分析现有实现（代码、原型、系统）并生成
适当的设计或架构文档。在以下情况使用：
- 您在没有首先编写设计文档的情况下构建了功能
- 您继承了没有文档的代码库
- 您制作了需要正式化的机制原型
- 您需要记录现有代码背后的"为什么"

---

## 工作流程

## 阶段 1：解析参数

**格式**：`/reverse-document <type> <path>`

**类型选项**：
- `design` → 生成游戏设计文档（GDD 章节）
- `architecture` → 生成架构决策记录（ADR）
- `concept` → 从原型生成概念文档

**路径**：要分析的目录或文件
- `src/gameplay/combat/` → 所有战斗相关代码
- `src/core/event-system.cpp` → 特定文件
- `prototypes/stealth-mech/` → 原型目录

**示例**：
```bash
/reverse-document design src/gameplay/magic-system
/reverse-document architecture src/core/entity-component
/reverse-document concept prototypes/vehicle-combat
```

## 阶段 2：分析实现

**读取并理解代码/原型**：

**对于设计文档（GDD）：**
- 识别机制、规则、公式
- 提取游戏数值（伤害、冷却时间、范围）
- 查找状态机、能力系统、进度
- 检测代码中处理的边界情况
- 映射依赖关系（哪些系统交互？）

**对于架构文档（ADR）：**
- 识别模式（ECS、单例、观察者等）
- 理解技术决策（线程、序列化等）
- 映射依赖关系和耦合
- 评估性能特征
- 查找约束和权衡

**对于概念文档（原型分析）：**
- 识别核心机制
- 提取涌现的游戏玩法模式
- 注意什么有效 vs 什么无效
- 查找技术可行性见解
- 记录玩家幻想/感觉

## 阶段 3：提出澄清问题

**不要**只描述代码。**询问**意图：

**设计问题**：
- "我看到一个 [资源] 系统，在 [活动] 期间耗尽。这是为了：
  - 节奏（防止垃圾邮件）？
  - 资源管理（战略深度）？
  - 还是其他？"
- "[机制] 似乎是核心。这是核心支柱，还是支持功能？"
- "[数值] 随 [因素] 指数增长。是故意的，还是需要重新平衡？"

**架构问题**：
- "您正在使用服务定位器模式。这是选择为了：
  - 可测试性（模拟依赖）？
  - 解耦（减少硬引用）？
  - 还是从现有代码继承的？"
- "我看到手动内存管理而不是智能指针。性能要求，还是遗留的？"

**概念问题**：
- "原型强调潜行而非战斗。这是预期的支柱吗？"
- "玩家似乎利用抓钩加速。是特性还是 bug？"

## 阶段 4：展示发现

在起草之前，展示您发现的：

```
我已分析 [路径]/。这是我发现的：

已实现的机制：
- [机制-a] 带有 [属性]（例如，时间窗口、冷却时间）
- [机制-b]（例如，两个状态之间的交互）
- [资源] 系统（在 [动作] 时耗尽，在 [条件] 时恢复）
- [状态] 系统（累积，触发 [效果]）

发现的公式：
- [输出] = [使用发现变量的公式]
- [次要输出] = [公式]

意图不明确的区域：
1. [资源] 系统 — 节奏还是资源管理？
2. [机制] — 核心支柱还是支持功能？
3. [数值] 缩放 — 有意设计还是需要调整？

在我起草设计文档之前，您能澄清这些要点吗？
```

等待用户在起草之前澄清意图。

## 阶段 5：使用模板起草文档

基于类型，使用适当的模板：

| 类型 | 模板 | 输出路径 |
|------|----------|-------------|
| `design` | `templates/design-doc-from-implementation.md` | `design/gdd/[system-name].md` |
| `architecture` | `templates/architecture-doc-from-code.md` | `docs/architecture/[decision-name].md` |
| `concept` | `templates/concept-doc-from-prototype.md` | `prototypes/[name]/CONCEPT.md` 或 `design/concepts/[name].md` |

**草案结构**：
- 捕获**存在什么**（机制、模式、实现）
- 记录**为什么存在**（与用户澄清的意图）
- 识别**缺少什么**（未处理的边界情况、设计差距）
- 标记**后续工作**（平衡调整、缺失功能）

## 阶段 6：展示草案并请求批准

**协作协议**：
```
我已根据您的代码和澄清起草了 [system-name] 设计文档。

[展示关键章节：概述、机制、公式、设计意图]

我添加的内容：
- 按您的澄清将 [机制] 记录为"[意图]"
- 添加了代码中不存在的边界情况（例如，如果 [资源] 在 [动作] 中间达到 0 会发生什么？）
- 标记了平衡问题：[缩放类型] 缩放在 [边界条件]

标记为不完整的章节：
- "[系统] 与 [其他系统] 的交互"（尚未完全实现）
- "[变体或功能]"（目前仅实现了 [子集]）

我可以将其写入 design/gdd/[system-name].md 吗？
```

等待批准。用户可能在写入之前请求更改。

## 阶段 7：使用元数据写入文档

获得批准后，用特殊标记写入文件：

```markdown
---
status: reverse-documented
source: [path/]
date: [today]
verified-by: [用户姓名]
---

# [系统名称] 设计

> **注意**：此文档是从现有实现反向工程而来的。
> 它捕获了当前行为和澄清的设计意图。某些章节可能不完整，
> 因为实现是部分的或意图不清楚。

[文档其余部分...]
```

## 阶段 8：标记后续工作

写入后，建议后续步骤：

```
✅ 已写入 design/gdd/combat-system.md

推荐跟进：
1. 在战斗公式上运行 /balance-check（指数缩放问题）
2. 为耐力系统架构决策创建 ADR
3. 实现缺失的边界情况：
   - 耐力在连击中耗尽的行为
   - 闪避期间的眩晕状态
4. 当武器种类实现时扩展设计文档

您想现在处理这些吗？
```

## 模板选择逻辑

| 如果分析... | 使用模板... | 因为... |
|----------------|-----------------|------------|
| `src/gameplay/*` | design-doc-from-implementation.md | 游戏玩法机制 → GDD |
| `src/core/*`, `src/ai/*` | architecture-doc-from-code.md | 核心系统 → ADR |
| `prototypes/*` | concept-doc-from-prototype.md | 实验 → 概念文档 |
| `src/networking/*` | architecture-doc-from-code.md | 技术系统 → ADR |
| `src/ui/*` | design-doc-from-implementation.md | UI/UX → 设计规范 |

## 协作协议

此 skill 遵循协作设计原则：

1. **先分析**：读取代码，理解实现
2. **询问意图**：询问"为什么"，而不仅仅是"什么"
3. **展示发现**：展示发现，突出不清楚的区域
4. **用户澄清**：将意图与意外分开
5. **起草文档**：基于现实 + 意图创建文档
6. **展示草案**：显示关键章节，解释添加的内容
7. **获取批准**："我可以写入 [filepath] 吗？" 批准时：裁决：**完成** — 文档已生成。拒绝时：裁决：**阻塞** — 用户拒绝写入。
8. **标记跟进**：建议相关工作，不要自动执行

**永远不要假设意图。在记录"为什么"之前始终询问。**
