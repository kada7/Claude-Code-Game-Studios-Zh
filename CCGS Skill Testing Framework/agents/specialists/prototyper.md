# Agent 测试规范：prototyper

## Agent 概述
- **领域**：在 `prototypes/` 目录中进行快速可丢弃的原型、概念验证实验、机制可行性测试。标准有意放宽以提高速度——原型不是生产代码。
- **不负责**：`src/` 中的生产源代码（由 gameplay-programmer 负责）、设计文档（由 game-designer 负责）、生产级架构决策（由 lead-programmer / technical-director 负责）
- **模型层级**：Sonnet
- **阶段门 ID**：无；在原型结束后生成推荐文档；不参与阶段门

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用快速原型、prototypes/ 目录、可丢弃代码）
- [ ] `allowed-tools:` 列表与 Agent 角色匹配（读写范围限定在 prototypes/ 目录；无生产 src/ 写入权限）
- [ ] 模型层级为 Sonnet（专家的默认设置）
- [ ] Agent 定义明确说明原型代码不是生产代码，不得复制到 src/

---

## 测试用例

### 用例 1：领域内请求——原型一个抽卡机制
**输入**："在 2 小时内原型一个抽卡机制。核心问题：每回合抽 3 张牌，手牌上限为 7，这样的感觉好吗？我今天需要一个可以在试玩中测试的东西。"
**预期行为**：
- 生成一个用项目引擎脚本语言编写的最小可行原型，范围限定在 `prototypes/card-draw-mechanic/`
- 代码优先考虑速度而非正确性：不需要单元测试，不需要文档注释，原型中允许使用全局状态
- 实现最小可行机制：一个牌库、一个抽牌函数（抽 N 张牌）、一个带大小限制的手牌容器，以及一个简单的 UI 或调试输出来验证状态
- 不实现生产模式（依赖注入、信号、数据驱动配置），除非使用它们比不使用花费更少时间
- 在原型文件夹中包含一个 `README.md`：正在测试的假设、如何运行、在试玩中观察什么

### 用例 2：领域外请求——生产级实现
**输入**："抽卡机制原型效果很好。现在为 src/gameplay/cards/ 编写卡片系统的生产实现。"
**预期行为**：
- 不向 `src/` 写入生产代码
- 明确声明："prototyper 在 prototypes/ 中生成可丢弃代码以验证概念；已验证机制的生产实现由 gameplay-programmer 处理"
- 提供生成过渡文档：原型证明了什么、生产实现应保留什么（机制）、应丢弃什么（可丢弃的实现模式）
- 不将原型代码复制到 src/ 中，也不建议将其作为起点而不警告其非生产质量

### 用例 3：原型验证机制——推荐输出
**输入**："抽卡原型试玩效果良好。三个会话都享受每回合抽 3 张牌、手牌上限 7 的设定。未观察到困惑。接下来怎么办？"
**预期行为**：
- 在 `prototypes/card-draw-mechanic/conclusion.md`（或等效位置）生成原型结论文档
- 文档包括：测试的假设、试玩方法（会话、时长、观察者笔记）、结果裁决（已验证）、关键发现（什么有效、观察到的任何小问题）、生产推荐（要保留的特定机制参数：每回合 3 张牌、手牌上限 7），以及将生产实现请求路由到 gameplay-programmer 的标志
- 不开始编写生产代码
- 输出结构化为决策就绪的推荐，而非叙述性摘要

### 用例 4：原型显示机制不可行——放弃说明
**输入**："基于物理的撬锁机制原型已完成。经过 4 次试玩会话，所有测试者都感到沮丧——需要太多精度，不好玩。一名测试者愤怒退出。"
**预期行为**：
- 在 `prototypes/lock-picking-physics/conclusion.md` 生成原型放弃说明
- 文档包括：测试的假设、结果裁决（已放弃）、具体原因（精度门槛太高、负面情绪反应、愤怒退出事件作为证据），以及探索替代方法的建议（简化的锁芯机制、基于节奏的替代方案、完全移除该机制）
- 不推荐因沉没成本而坚持原型机制
- 不将结果标记为不确定——在 4 次会话且一致负面回应后，放弃是正确的裁决

### 用例 5：上下文传递——使用项目的引擎脚本语言
**输入上下文**：项目使用 Godot 4.6 和 GDScript（在 technical-preferences.md 中配置）。
**输入**："原型一个基本的网格移动系统——玩家点击一个格子，角色移动到该位置。"
**预期行为**：
- 用 GDScript 生成原型——不是 Python、C# 或伪代码
- 使用适用于网格的 Godot 4.6 节点类型：TileMap 或自定义网格管理器节点，CharacterBody2D 或 Node2D 用于玩家
- 不应用生产编码标准（不需要测试覆盖率、不需要文档注释、允许全局状态）
- 将输出写入 `prototypes/grid-movement/` 而非 `src/`
- 如果 Godot 4.6 API 不确定（考虑到 VERSION.md 中注明的 LLM 知识截止日期），用注释标记特定 API 以根据 Godot 4.6 文档进行验证

## 协议合规性

- [ ] 保持在声明的领域内（仅限 prototypes/ 目录；用于概念验证的可丢弃代码）
- [ ] 将生产实现请求重定向到 gameplay-programmer，并提供过渡文档
- [ ] 在原型评估后生成结构化结论文档（已验证或已放弃裁决）
- [ ] 不推荐在不明确警告的情况下以生产形式保留原型代码
- [ ] 使用项目配置的引擎和脚本语言；标记版本不确定性

---

## 覆盖范围说明
- 用例 2（生产重定向）至关重要——原型代码泄漏到 src/ 是常见的质量问题
- 用例 4（放弃诚实性）测试 Agent 是否避免沉没成本偏差——失败的原型应干净利落地放弃
- 用例 5 要求 technical-preferences.md 已配置引擎和语言；如果未配置，测试不完整
- 编码标准的有意放宽是一个特性，而非缺陷——不要在原型输出中将缺少测试或文档注释标记为失败
- 无自动运行器；通过手动或 `/skill-test` 进行审查
