## 阶段 1: 解析参数

读取参数以获取目标版本或冲刺编号。如果给定版本，使用相应的 git 标签。如果给定冲刺编号，使用冲刺日期范围。

验证仓库是否已初始化: 运行 `git rev-parse --is-inside-work-tree` 以确认 git 可用。如果不是 git 仓库，通知用户并优雅地中止。

---

## 阶段 2: 收集变更数据

读取自上次标签或发布以来的 git 日志:

```
git log --oneline [last-tag]..HEAD
```

如果没有标签存在，读取完整日志或合理的最近范围 (最近 100 个提交)。

从 `production/sprints/` 读取相关时期的冲刺报告，以了解计划工作背后的变化和上下文。

从 `design/gdd/` 读取此期间实现的任何新功能的已完成设计文档。

---

## 阶段 3: 分类变更

将每个变更分类到以下类别之一:

- **新功能**: 全新的游戏系统、模式或内容
- **改进**: 对现有功能的增强、UX 改进、性能提升
- **Bug 修复**: 对损坏行为的更正
- **平衡变更**: 游戏数值、难度、经济的调整
- **已知问题**: 团队已知但尚未解决的问题
- **其他**: 不符合上述类别的变更，或消息太模糊无法自信分类的提交

对于每个提交，检查消息是否包含任务 ID 或 story 引用
(例如 `[STORY-123]`, `TR-`, `#NNN`, 或类似)。计算缺少任何任务引用的提交数量
并在阶段 4 指标部分将其作为: `Commits without task reference: [N]` 包含。

## 阶段 4: 生成内部更新日志

```markdown
# 内部更新日志: [Version]
日期: [Date]
冲刺: [覆盖的冲刺编号]
提交: [Count] ([first-hash]..[last-hash])

## 新功能
- [Feature Name] -- [技术描述、受影响的系统]
  - 提交: [hash1], [hash2]
  - 负责人: [who implemented it]
  - 设计文档: [link if applicable]

## 改进
- [Improvement] -- [技术上更改了什么及原因]
  - 提交: [hashes]
  - 负责人: [who]

## Bug 修复
- [BUG-ID] [Bug 和根本原因描述]
  - 修复: [What was changed]
  - 提交: [hashes]
  - 负责人: [who]

## 平衡变更
- [What was tuned] -- [Old value -> New value] -- [Design intent]
  - 负责人: [who]

## 技术债务 / 重构
- [清理了什么及原因]
  - 提交: [hashes]

## 其他
- [不符合其他类别的变更，或模糊的提交消息]
  - 提交: [hashes]

## 已知问题
- [Issue description] -- [Severity] -- [ETA for fix if known]

## 指标
- 总提交数: [N]
- 更改的文件: [N]
- 添加的行数: [N]
- 删除的行数: [N]
- 没有任务引用的提交: [N]
```

## 阶段 5: 生成面向玩家的更新日志

```markdown
# [Version] 新内容

## 新功能
- **[Feature Name]**: [玩家友好的描述，说明他们现在可以做什么
  以及为什么令人兴奋。关注体验，而非实现。]

## 改进
- **[What improved]**: [这如何让游戏对玩家更好。
  具体但避免行话。]

## Bug 修复
- 修复了 [描述玩家遇到的情况，而非代码中的错误]
- 修复了 [玩家可见的症状]

## 平衡变更
- [玩家易懂术语中的更改及设计意图。
  示例: "治疗药水现在恢复 50 HP (从 30 提升) -- 我们觉得
  玩家在后期游戏遭遇战中需要更多恢复选项。"]

## 已知问题
- 我们已知 [玩家术语中的问题描述] 并正在努力
  修复。 [Workaround if one exists.]

---
感谢您的游玩! 您的反馈帮助我们让游戏变得更好。
在 [link] 报告问题。
```

---

## 阶段 6: 输出

向用户输出两个更新日志。内部更新日志是主要工作文档。面向玩家的更新日志在审核后可用于社区发布。

---

## 阶段 7: 提供文件写入

展示更新日志后，询问用户:

> "我可以将此更新日志写入 `docs/CHANGELOG.md` 吗?
> [A] 是的，附加此条目 (如果文件已存在则推荐)
> [B] 是的，完全覆盖文件
> [C] 不 — 我将手动复制"

- 询问前检查 `docs/CHANGELOG.md` 是否存在。如果存在，将
  推荐默认设置为 **[A] append**。
- 如果用户选择 [A]: 将新的内部更新日志条目附加到现有文件的顶部 (最新的条目在前)。
- 如果用户选择 [B]: 用新更新日志覆盖文件。
- 如果用户选择 [C]: 停止而不写入。

成功写入后: 裁决: **CHANGELOG WRITTEN** — 更新日志已保存到 `docs/CHANGELOG.md`。
如果用户拒绝: 裁决: **COMPLETE** — 更新日志已生成。

---

## 阶段 7: 后续步骤

- 使用 `/patch-notes [version]` 生成用于公开发布的样式化、保存版本。
- 在外部发布更新日志前使用 `/release-checklist`。

### 指南

- 切勿在面向玩家的更新日志中暴露内部代码引用、文件路径或开发者名称
- 将相关变更组合在一起，而非列出单个提交
- 如果提交消息不清楚，检查相关文件和冲刺数据以获取上下文
- 平衡变更应始终包括设计理由，而非仅数字
- 已知问题应诚实 — 玩家欣赏透明度
- 如果 git 历史混乱 (合并提交、还原、修正提交)，清理叙述而非逐字列出每个提交
