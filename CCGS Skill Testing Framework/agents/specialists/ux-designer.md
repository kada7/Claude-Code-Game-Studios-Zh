# Agent Test Spec: ux-designer

## Agent 摘要
领域：用户体验流程、交互设计、信息架构、输入处理设计和新手引导 UX。
不负责：视觉艺术风格（art-director）、UI 实现代码（ui-programmer）。
模型层级：Sonnet（默认）。
未分配 gate ID。

---

## 静态断言（结构）

- [ ] `description:` 字段存在且具有领域特定性（引用 UX 流程/交互设计/信息架构）
- [ ] `allowed-tools:` 列表包含 Read, Write, Edit, Glob, Grep
- [ ] 模型层级为 Sonnet（专家默认）
- [ ] Agent 定义不声称对视觉艺术指导或 UI 实现代码拥有权限

---

## 测试用例

### 用例 1：领域内请求 - 适当的输出
**输入：** "为生存游戏设计库存管理流程。"
**预期行为：**
- 生成库存的用户流程图（状态和转换）：打开、浏览、选择物品、子操作（装备/丢弃/组合）、关闭
- 定义所有交互状态（默认、悬停、选中、空槽位、锁定槽位）
- 指定每个操作的输入映射（键盘、游戏手柄如适用）
- 注明认知负荷考虑因素（例如，无需滚动即可查看的最大物品数）
- 不生成视觉设计（颜色、图标）或实现代码

### 用例 2：领域外请求 - 正确重定向
**输入：** "使用 GDScript 实现具有拖放支持的库存屏幕。"
**预期行为：**
- 不生成实现代码
- 明确说明 UI 代码实现属于 `ui-programmer`
- 将请求重定向至 `ui-programmer`
- 注明应将 UX 流程规范提供给 ui-programmer 作为实现参考

### 用例 3：流程深度冲突 - 简化
**输入：** "主设计师表示当前的 5 步制作流程太深；最多允许 3 步。"
**预期行为：**
- 生成一个修订后的 3 步流程，合并原始的 5 步序列
- 清晰展示合并或删除了哪些内容，以及从可用性角度为什么每个合并是安全的
- 不简单地删除步骤而不处理每个被删除步骤的用户目标
- 如果 3 步约束使任何必需用例变得不可能，则标记并提出替代方案

### 用例 4：无障碍性冲突
**输入：** "新手引导流程使用定时提示（3 秒后自动前进）以保持节奏，但这与用户控制时间的无障碍性要求冲突。"
**预期行为：**
- 识别与 WCAG 2.1 2.2.1（可调节时间）的冲突
- 不为了保持节奏而覆盖无障碍性要求
- 与 `accessibility-specialist` 协调以商定合规解决方案
- 提出替代方案：悬停暂停、跳过按钮、禁用自动前进的设置选项

### 用例 5：上下文传递 - 玩家心智模型研究
**输入：** 在上下文中提供的游戏测试研究："玩家一致期望'制作'选项在库存屏幕内，而不是在单独的顶级菜单中。" 请求："重新设计制作的导航信息架构。"
**预期行为：**
- 引用研究中的具体玩家期望（制作预期在库存内）
- 重新组织信息架构，将制作作为库存屏幕内的标签页或面板
- 不生成与所述玩家心智模型相矛盾的设计，除非有明确理由
- 在设计决策的理由中注明研究来源

---

## 协议合规性

- [ ] 保持在声明的领域内（UX 流程、交互设计、信息架构、新手引导）
- [ ] 将代码实现重定向至 ui-programmer，视觉风格重定向至 art-director
- [ ] 返回结构化发现（状态图、流程步骤、输入映射）而非自由形式的意见
- [ ] 当流程有时间或认知负荷约束时，与 accessibility-specialist 协调
- [ ] 基于提供的用户研究设计流程，而非假设行为
- [ ] 根据用户目标记录流程决策的理由

---

## 覆盖范围说明
- 库存流程（用例 1）应写入 `design/ux/` 作为 ui-programmer 实现的规范
- 心智模型用例（用例 5）验证 Agent 应用研究证据而非直觉
- 无障碍性协调（用例 4）确认 Agent 不会为了 UX 美学而覆盖无障碍性要求
