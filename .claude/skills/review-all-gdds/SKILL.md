# 审查所有 GDD

此 skill 同时读取每个系统 GDD 并执行两种互补的
无法通过单独审查每个 GDD 完成的审查：

1. **跨 GDD 一致性** — 文档之间的矛盾、陈旧引用和所有权
   冲突
2. **游戏设计整体论** — 只有在看到所有系统时才会出现的问题：
   主导策略、破碎的经济、认知过载、支柱
   漂移、竞争的进度循环

**这与 `/design-review` 不同**，后者审查一个 GDD 的内部
完整性。此 skill 审查所有 GDD 之间的*关系*。

**何时运行：**
- 在所有 MVP 层级 GDD 单独获得批准后
- 生产中任何 GDD 被显著修订后
- 在 `/create-architecture` 开始之前（建立在不一致 GDD 上的架构
  会继承这些不一致）

**参数模式：**

**焦点：** `$ARGUMENTS[0]`（空白 = `full`）

- **无参数 / `full`**：一致性和设计理论都通过
- **`consistency`**：仅跨 GDD 一致性检查（更快）
- **`design-theory`**：仅游戏设计整体论检查
- **`since-last-review`**：仅自上次审查报告以来修改的 GDD（基于 git）

## 阶段 1：加载所有内容

### 阶段 1a — L0：摘要扫描（快速，低 token）

在读取任何完整文档之前，使用 Grep 从所有 GDD 文件提取 `## Summary` 章节：

```
Grep pattern="## Summary" glob="design/gdd/*.md" output_mode="content" -A 5
```

向用户展示清单：
```
找到 [N] 个 GDD。摘要：
  • combat.md — [摘要文本]
  • inventory.md — [摘要文本]
  ...
```

对于 `since-last-review` 模式：运行 `git log --name-only` 识别自上次审查报告文件写入以来修改的 GDD。在任何完整读取之前向用户展示基于摘要的范围内 GDD。仅对这些 GDD 加上其"关键依赖"中列出的任何 GDD 进行到 L1。

### 阶段 1b — 注册表预加载（快速基线）

在完整读取任何 GDD 之前，检查实体注册表：

```
Read path="design/registry/entities.yaml"
```

如果注册表存在并有条目，将其用作**预构建的冲突
基线**：已知实体、物品、公式和常量及其
权威值和源 GDD。在阶段 2 中，首先在 GDD 中全局搜索注册的名称 — 这比在知道要查找什么之前完整读取所有 GDD 更快。

如果注册表为空或不存在：在没有它的情况下继续。在报告中记录：
"实体注册表为空 — 一致性检查仅依赖完整 GDD 读取。
在此审查之后运行 `/consistency-check` 以填充注册表。"

### 阶段 1c — L1/L2：完整文档加载

完整读取范围内的文档：

1. `design/gdd/game-concept.md` — 游戏愿景、核心循环、MVP 定义
2. 如果存在 `design/gdd/game-pillars.md` — 设计支柱和反支柱
3. `design/gdd/systems-index.md` — 权威系统列表、层级、依赖关系、状态
4. **`design/gdd/` 中的每个范围内系统 GDD** — 完整读取（跳过
   game-concept.md 和 systems-index.md — 那些在上面已读取）

报告："已加载 [N] 个系统 GDD，涵盖 [M] 个系统。支柱：[列表]。反支柱：[列表]。"

如果存在的系统 GDD 少于 2 个，停止：
> "跨 GDD 审查需要至少 2 个系统 GDD。先编写更多 GDD，
> 然后重新运行 `/review-all-gdds`。"

### 并行执行

阶段 2（一致性）和阶段 3（设计理论）是独立的 — 它们读取相同的 GDD 输入但产生单独的报告。将两者作为并行 Task 代理同时生成，而不是等待阶段 2 完成后再开始阶段 3。在写入合并报告之前收集两个结果。

## 阶段 2：跨 GDD 一致性

遍历每对和每组 GDD 以查找矛盾和差距。

### 2a：依赖双向性

对于每个 GDD 的依赖章节，检查每个列出的依赖是否是
相互的：
- 如果 GDD-A 列出"依赖于 GDD-B"，检查 GDD-B 是否将 GDD-A 列为依赖者
- 如果 GDD-A 列出"被 GDD-C 依赖"，检查 GDD-C 是否将 GDD-A 列为依赖
- 将任何单向依赖标记为一致性问题

```
⚠️  依赖不对称
[system-a].md 列出：依赖于 → [system-b].md
[system-b].md 没有将 [system-a].md 列为依赖者
→ 其中一个文档有过时的依赖章节
```

### 2b：规则矛盾

对于在任何 GDD 中定义的每个游戏规则、机制或约束，检查是否有
任何其他 GDD 为相同情况定义了矛盾的规则：

要扫描的类别：
- **下限/上限规则**：任何 GDD 是否为输出定义最小值？其他文档是否说不同系统可以绕过该下限？这些矛盾。
- **资源所有权**：如果两个 GDD 都定义共享资源如何累积或消耗，它们是否一致？
- **状态转换**：如果 GDD-A 描述角色死亡时会发生什么，
  GDD-B 对同一事件的描述是否一致？
- **时间**：如果 GDD-A 说"X 在同一帧发生"，GDD-B 是否假设
  它异步发生？
- **堆叠规则**：如果 GDD-A 说状态效果堆叠，GDD-B 是否假设
  它们不堆叠？

```
🔴 规则矛盾
[system-a].md："减少后的最小 [输出] 是 [floor_value]"
[system-b].md："[机制] 绕过 [system-a] 的规则，可以将 [输出] 减少到 0"
→ 这些规则直接矛盾。哪个 GDD 是权威的？
```

### 2c：陈旧引用

对于每个跨文档引用（GDD-A 提到来自 GDD-B 的机制、数值或
系统名称），验证被引用的元素是否仍以相同的名称和行为存在于 GDD-B 中：

- 如果 GDD-A 说"战斗系统的连击倍数进入分数"，检查
  战斗 GDD 是否确实定义了输出到分数的连击倍数
- 如果 GDD-A 引用"[system].md 中定义的进度曲线"，检查
  [system].md 是否确实有该曲线，而不是不同的进度模型
- 如果 GDD-A 在 GDD-B 之前编写并假设了 GDD-B 后来
  不同设计的机制，将 GDD-A 标记为包含陈旧引用

```
⚠️  陈旧引用
inventory.md（先编写）："物品重量使用来自 movement.md 的负重公式"
movement.md（后编写）：没有定义负重公式 — 使用固定
  携带限制
→ inventory.md 引用了一个不存在的公式
```

### 2d：数据和调整旋钮所有权冲突

两个 GDD 不应该都声称拥有相同的数据或调整旋钮。扫描所有
跨所有 GDD 的调整旋钮章节并标记重复：

```
⚠️  所有权冲突
[system-a].md 调整旋钮："[multiplier_name] — 控制 [输出] 缩放"
[system-b].md 调整旋钮："[multiplier_name] — 使用 [factor] 缩放 [输出]"
→ 两个 GDD 定义同一输出的倍数。哪个拥有最终值？
  这将产生双重应用错误或设计冲突。
```

### 2e：公式兼容性

对于公式相连的 GDD（一个的输出输入另一个），
检查上游公式的输出范围是否在下游公式的预期输入范围内：

- 如果 [system-a].md 输出 [min]–[max] 之间的值，而 [system-b].md 设计为接收 [min2]–[max2] 之间的值，不匹配是故意的吗？
- 如果经济 GDD 期望范围 X 的资源获取，而进度 GDD 在范围 Y 生成它，经济将变得微不足道或无法访问 — 这是预期的吗？

将不兼容性标记为顾虑（需要设计判断，不一定是错误的）：

```
⚠️  公式范围不匹配
[system-a].md：最大 [输出] = [value_a]（在最大 [条件] 时）
[system-b].md：基础 [输入] = [value_b]，最大 [输入] = [value_c]
→ 后期-[阶段] [场景] 可以在单个 [事件] 中解决。
  这是故意的吗？如果不是，需要调整 [system-a] 的上限或 [system-b] 的上限。
```

### 2f：验收标准交叉检查

扫描所有 GDD 的验收标准章节以查找矛盾：

- GDD-A 标准："玩家不能死于单次攻击"
- GDD-B 标准："Boss 攻击造成玩家最大生命值的 150%"
这些验收标准不能同时通过。

## 阶段 3：游戏设计整体论

通过游戏设计理论和玩家心理学的镜头一起审查所有 GDD。这些是单个 GDD 审查无法捕捉的问题，因为它们需要同时看到所有系统。

### 3a：进度循环竞争

游戏应该有一个主导的进度循环，玩家感觉它是游戏的"要点"，有支持循环为其提供养分。当多个系统平等竞争作为主要进度驱动时，玩家不知道游戏是关于什么的。

扫描所有 GDD 以查找以下系统：
- 奖励玩家的主要资源（XP、等级、声望、解锁）
- 将自己定义为"核心"或"主要"循环
- 与其他系统做相同的事情具有可比较的深度和时间投入

```
⚠️  竞争的进度循环
combat.md：奖励 XP、解锁能力，被描述为"核心循环"
crafting.md：奖励 XP、解锁配方，被描述为"主要活动"
exploration.md：奖励 XP、解锁地图区域，被描述为"主要驱动"
→ 三个系统都声称是主要进度循环，都奖励相同的主要货币。玩家将优化一个并忽略其他。
  考虑：一个主要循环，其他作为支持系统。
```

### 3b：玩家注意力预算

计算在典型会话期间同时需要玩家主动注意的系统数量。每个主动管理的系统都消耗注意力：

- 主动 = 玩家必须在游戏过程中定期对此系统做出决策
- 被动 = 系统自动运行，玩家看到结果但不管理它

超过 3-4 个同时主动的系统会为大多数玩家创造认知过载。展示计数，如果超过 4 个并发主动系统则标记：

```
⚠️  认知负荷风险
在 [核心循环时刻] 同时主动的系统：
  1. [system-a].md — [决策类型]（主动）
  2. [system-b].md — [资源管理]（主动）
  3. [system-c].md — [跟踪]（主动）
  4. [system-d].md — [物品/动作使用]（主动）
  5. [system-e].md — [冷却/计时器管理]（主动）
  6. [system-f].md — [协调决策]（主动）
→ 核心循环期间有 6 个同时主动的系统。
  研究表明大多数玩家的舒适限制是 3-4 个。
  考虑：其中哪些可以变为被动或简化？
```

### 3c：主导策略检测

主导策略使其他策略无关紧要 — 玩家发现它，专门使用它，觉得游戏其余部分很无聊。查找：

- **资源垄断**：一种策略生成资源的速度显著快于所有其他策略
- **无风险力量**：既是高奖励又是低风险的策略
  （如果存在高风险策略，它们需要成比例更高的奖励）
- **没有权衡**：在所有维度上都优于所有其他选项的选项
- **明显的最优路径**：如果任何进度选择是"明显正确的"，
  其他选择就不是真正的选择

```
⚠️  潜在主导策略
combat.md：远程攻击造成 80% 的近战伤害且没有风险
combat.md：近战攻击造成 100% 伤害但需要近距离
→ 除非近战有显著的补偿优势（AOE、眩晕、
  资源恢复），否则远程是主导的 — 更高的安全性，仅少 20%
  伤害。考虑近战提供的远程无法提供的东西。
```

### 3d：经济循环分析

识别所有 GDD 中的资源（金币、XP、制作材料、耐力、健康、法力等）。对于每个资源，映射其**来源**（玩家如何获得它）和**汇点**（玩家如何花费它）。

标记危险的经济状况：

| 状况 | 迹象 | 风险 |
|-----------|------|------|
| **无限来源，没有汇点** | 资源无限累积 | 后期游戏变得微不足道地容易 |
| **汇点，没有来源** | 资源耗尽到零 | 系统变得不可用 |
| **来源 >> 汇点** | 盈余累积 | 资源变得无意义 |
| **汇点 >> 来源** | 持续稀缺 | 沮丧和门槛限制 |
| **正反馈循环** | 更多资源 → 更容易获得更多 | 失控领先，滚雪球 |
| **没有追赶机制** | 落后加速赤字 | 无法恢复的状态 |

```
🔴 经济失衡：无界正反馈
金币经济：
  来源：怪物掉落（随玩家能力缩放），商人出售（无限）
  汇点：装备购买（一次性），能力升级（有限数量）
→ 购买装备和能力后，金币没有汇点。
  无限盈余。金币在游戏中期变得无意义。
  添加持续的金币汇点（维护费、消耗品、外观、赌博）。
```

### 3e：难度曲线一致性

当多个系统随玩家进度缩放时，它们必须以兼容的方向和兼容的速率缩放。不匹配的缩放曲线会产生意外的难度峰值或变得微不足道。

对于每个随时间缩放的系统，提取：
- 什么缩放（敌人生命、玩家伤害、资源成本、区域大小）
- 如何缩放（线性、指数、阶梯式）
- 何时缩放（等级、时间、区域）

比较所有缩放曲线。标记不匹配：

```
⚠️  难度曲线不匹配
combat.md：敌人生命随区域指数缩放（每区域 ×2）
progression.md：玩家伤害随等级线性缩放（每等级 +10%）
→ 到区域 5，敌人有 32 倍基础生命；玩家造成约 1.5 倍基础伤害。
  差距无限扩大。后期区域将变得无法访问地困难，除非曲线协调一致。
```

### 3f：支柱对齐

每个系统应该清楚地服务于至少一个设计支柱。不服务于任何支柱的系统是"设计范围蔓延" — 它在游戏中但不在服务游戏试图成为的东西。

对于每个 GDD 系统，将其玩家幻想章节与设计支柱进行对比。
标记任何陈述的幻想不映射到任何支柱的系统：

```
⚠️  支柱漂移
fishing-system.md：玩家幻想 — "平静、冥想的活动"
支柱："残酷战斗"、"紧张生存"、"涌现故事"
→ 钓鱼系统不服务于三个支柱中的任何一个。要么添加一个涵盖它的支柱，
  重新设计它以服务于现有支柱，要么削减它。
```

还要检查反支柱 — 标记任何做反支柱明确说游戏不会做的事情的系统：

```
🔴 反支柱违规
反支柱："我们不会有线性故事进度 — 玩家定义他们的路径"
main-quest.md：定义了一个 12 章的线性故事，有强制顺序
→ 此系统直接违反定义的反支柱。
```

### 3g：玩家幻想连贯性

所有系统的玩家幻想应该是兼容的 — 它们应该强化玩家在此游戏中是什么的一致身份。冲突的玩家幻想造成身份混乱。

```
⚠️  玩家幻想冲突
combat.md："你是一个无情、精准的战士 — 每次击杀都是应得的"
dialogue.md："你是一个魅力十足的外交官 — 暴力总是可以避免的"
exploration.md："你是一个鲁莽的冒险家 — 毫无计划地投入"
→ 三个系统呈现不兼容的身份。玩家会觉得游戏不知道它希望他们成为什么。考虑：这些幻想是否从同一核心身份的不同角度服务，还是它们真正冲突？
```

## 阶段 4：跨系统场景演练

从玩家的视角逐步浏览游戏，以查找只有在多个系统的交互边界才会出现的问题 — 静态分析单个 GDD 无法浮现的事情。

### 4a：识别关键多系统时刻

扫描所有 GDD 并识别 3–5 个最重要的面向玩家的时刻，其中多个系统同时激活。特别查找：

- **战斗 + 经济重叠**：杀死掉落资源的敌人、在战斗中花费资源、死亡/重生与经济状态交互
- **进度 + 难度重叠**：战斗中升级、能力解锁改变战斗可行性、进度里程碑处的难度缩放
- **叙事 + 游戏玩法重叠**：对话选择锁定/解锁机制、故事节拍中断资源循环、任务完成触发系统状态变化
- **3+ 系统链**：任何触发系统 A 的玩家动作，A 输入系统 B，触发系统 C（这些是最高风险的交互路径）

在继续之前列出每个识别的场景及其单行描述。

### 4b：逐步演练每个场景

对于每个场景，明确逐步浏览：

1. **触发器** — 什么玩家动作或游戏事件启动这个？
2. **激活顺序** — 哪些系统激活，以什么顺序？
3. **数据流** — 每个系统输出什么，该输出是否是链中下一个系统的有效输入？
4. **玩家体验** — 玩家在每一步看到、听到或感觉到什么？
5. **故障模式** — 是否有以下任何情况？
   - **竞态条件**：两个系统试图同时修改相同状态
   - **反馈循环**：系统 A 放大系统 B 再重新放大系统 A，没有上限或阻尼器
   - **破损状态转换**：系统假设前一个系统可能已改变的状态（例如，在战斗步骤可能导致死亡的"玩家活着"假设之后）
   - **矛盾消息**：玩家从对同一事件做出反应的两个系统收到冲突反馈（例如，"成功"声音 + "失败" UI）
   - **复合难度峰值**：两个系统在同一进度点都放大，将预期的难度增加相乘
   - **奖励冲突**：两个系统都对同一触发器做出反应，奖励一起超过预期值（双重获取）
   - **未定义行为**：GDD 没有指定在此组合状态下会发生什么（两个系统的规则都没有涵盖它）

```
示例演练：
场景：玩家在活跃任务期间在升级阈值处杀死精英敌人

触发器：玩家对精英敌人造成击杀
→ combat.md：奖励击杀 XP（100 点）
→ progression.md：XP 总计跨越等级阈值 → 触发升级
  输出：新等级、属性增加、能力解锁弹窗
→ quest.md：击杀计数标准满足 → 触发任务完成事件
  输出：任务奖励 XP（500 点）、完成音效
→ progression.md（再次）：添加任务 XP → 在同一帧触发第二次升级
  ⚠️  数据流问题：quest.md 在不检查是否已有升级的情况下奖励 XP。
  progression.md 没有针对并发升级事件的防护。未定义行为：玩家升级一次还是两次？
  能力弹窗是否触发两次？第二次升级使用更新前还是更新后的属性基线？
```

### 4c：标记场景问题

对于演练期间发现的每个问题，分类严重程度：

- **阻塞器**：未定义行为、破损状态转换或矛盾的
  玩家消息 — 此场景中的体验是破损或不相干的
- **警告**：复合峰值、没有上限的反馈循环、奖励冲突 — 
  体验有效但产生意外结果
- **信息**：轻微排序模糊或消息重叠 — 值得注意但不太可能引起玩家可见的问题

将所有发现添加到输出报告的**"跨系统场景问题"**下。
每个发现必须引用：场景名称、涉及的特定系统、问题发生的步骤，以及故障模式的性质。

## 阶段 7：交接

在构建选项之前，检查项目状态：

动态构建选项列表 — 仅包含适用的选项：

**选项池：**
- `[_] 应用快速修复：[W-XX 描述] 在 [gdd-name].md 中 — [工作量估计]`（每个简单编辑警告一个选项；仅用于警告级别，不是阻塞）

## 错误恢复协议

如果任何生成的代理返回阻塞、错误或未能完成：

1. **立即展示**：在继续之前报告"[代理名称]：阻塞 — [原因]"
2. **评估依赖关系**：如果被阻塞代理的输出被后期阶段需要，不要在没有用户输入的情况下越过该阶段
3. 通过 AskUserQuestion 提供三个选择：
   - 跳过此代理并在最终报告中记录差距
   - 使用更窄的范围重试（更少的 GDD、单系统焦点）
   - 在此停止并首先解决阻塞器
4. **始终产生部分报告** — 输出已完成的任何内容，以免工作丢失

## 协作协议

1. **静默读取** — 在展示任何内容之前加载所有 GDD
2. **展示所有内容** — 在要求任何操作之前呈现完整的一致性和设计理论分析
3. **区分阻塞与建议** — 不是每个问题都需要阻止架构；清楚哪些需要
4. **不要做出设计决策** — 标记矛盾和选项，但永远不要单方面决定哪个 GDD 是"正确的"
5. **写入前询问** — 在写入报告或更新系统索引之前确认
6. **具体** — 每个问题必须引用确切的 GDD、章节和涉及的文本；没有模糊的警告
