## 阶段 1：解析参数

解析审查模式（一次，存储供本次运行的所有门生成使用）：
1. 如果传入了 `--review [full|lean|solo]` → 使用该值
2. 否则读取 `production/review-mode.txt` → 使用该值
3. 否则 → 默认为 `lean`

查看 `.claude/docs/director-gates.md` 获取完整的检查模式。

确定模式：

- `new` → 生成空白游戏测试报告模板
- `analyze [path]` → 读取原始笔记并用结构化发现填充模板

---

## 阶段 2A：新建模板模式

生成此模板并输出给用户：

```markdown
# 游戏测试报告

## 会话信息
- **日期**：[日期]
- **构建**：[版本/提交]
- **时长**：[游戏时间]
- **测试员**：[姓名/ID]
- **平台**：[PC/主机/移动端]
- **输入方式**：[键鼠 / 手柄 / 触摸]
- **会话类型**：[首次 / 回归 / 针对性测试]

## 测试重点
[正在测试的特定功能或流程]

## 第一印象（前 5 分钟）
- **理解目标？** [是/否/部分]
- **理解控制？** [是/否/部分]
- **情绪反应**：[投入/困惑/无聊/沮丧/兴奋]
- **笔记**：[观察]

## 游戏流程
### 运作良好的方面
- [观察 1]

### 痛点
- [问题 1 — 严重程度：高/中/低]

### 困惑点
- [玩家困惑的地方及原因]

### 愉悦时刻
- [让玩家惊喜或满意的地方]

## 遇到的 Bug
| # | 描述 | 严重程度 | 可复现 |
|---|-------------|----------|-------------|

## 功能特定反馈
### [功能 1]
- **理解用途？** [是/否]
- **觉得有趣？** [是/否]
- **建议**：[测试员建议]

## 定量数据（如果有）
- **死亡次数**：[计数和位置]
- **每个区域的时间**：[细分]
- **使用的物品**：[什么和何时]
- **发现与遗漏的功能**：[列表]

## 总体评估
- **会再玩？** [是/否/可能]
- **难度**：[太简单 / 刚好 / 太难]
- **节奏**：[太慢 / 好 / 太快]
- **会话长度偏好**：[更短 / 好 / 更长]

## 本次会议的前 3 个优先事项
1. [最重要的发现]
2. [第二优先]
3. [第三优先]
```

## 阶段 2B：分析模式

读取所提供路径的原始笔记。与现有设计文档交叉引用。用结构化发现填充上述模板。标记任何与设计意图冲突的游戏测试观察。

---

## 阶段 3：行动路由

将所有发现分类到四个桶中：

- **需要设计变更** — 乐趣问题、玩家困惑、损坏的机制、与 GDD 预期体验冲突的观察
- **平衡调整** — 数值感觉不对、难度太陡或太平
- **Bug 报告** — 清晰的可复现实现缺陷
- **润色项** — 不阻塞进度，但后续需要处理的摩擦或感觉问题

展示分类列表，然后进行路由：

- **设计变更：** "在做出更改前，对受影响的设计文档运行 `/propagate-design-change [path]` 以找到下游影响。"
- **平衡调整：** "在调整数值前运行 `/balance-check [system]` 以验证完整的平衡情况。"
- **Bug：** "使用 `/bug-report` 正式跟踪这些。"
- **润色项：** "当团队到达该阶段时，将其添加到 `production/` 中的润色积压。"

---

## 阶段 3b：创意总监玩家体验审查

**审查模式检查** — 在生成 CD-PLAYTEST 之前应用：
- `solo` → 跳过。注意："CD-PLAYTEST 已跳过 — 单人模式。" 进入阶段 4（保存报告）。
- `lean` → 跳过（不是阶段门）。注意："CD-PLAYTEST 已跳过 — 精简模式。" 进入阶段 4（保存报告）。
- `full` → 正常生成。

分类发现后，使用门 **CD-PLAYTEST**（`.claude/docs/director-gates.md`）通过 Task 生成 `creative-director`。

传递：结构化报告内容、游戏支柱和核心幻想（来自 `design/gdd/game-concept.md`）、正在测试的特定假设。

在保存报告之前展示创意总监的评估。如果是顾虑或拒绝，向报告添加一个 `## 创意总监评估` 章节以捕获裁决和反馈。如果是批准，在报告中记录批准。

## 阶段 5：后续步骤

裁决：**完成** — 游戏测试报告已生成。

- 首先处理最高优先级的发现类别。
- 处理设计变更后：对更新的 GDD 重新运行 `/design-review`。
- 修复 bug 后：重新运行 `/bug-triage` 以更新优先级。
