## 阶段2：视觉识别基础（章节 1–4）

这四个章节定义了核心视觉语言。**所有其他章节都从它们流出。** 逐章节编写并写入文件，然后再进行下一个。

### 章节 1：视觉识别声明

**目标**：一行视觉规则加上 2–3 条支持原则，用于解决视觉歧义。

如果 game-concept.md 存在视觉锚点，呈现它并询问：
- "直接从此锚点构建？"
- "在扩展之前修订它？"
- "用新选项重新开始？"

**Agent 委托（强制）**：通过 Task 生成 `art-director`：
- 提供：游戏概念（电梯宣传语、核心幻想）、完整支柱集、平台目标、阶段1定框架中的任何参考游戏/艺术、如果存在的视觉锚点
- 询问："为此游戏起草视觉识别声明。提供：(1) 一行视觉规则，可以解决任何视觉决策的歧义；(2) 2–3 条支持视觉原则，每条都有一句设计测试（'当 X 模糊时，此原则说选择 Y'）。将所有原则直接锚定在已声明的支柱上 — 每条原则必须服务于特定支柱。"

向用户呈现 art-director 的草案。使用 `AskUserQuestion`：
- 选项：`[A] 确认这个` / `[B] 修改一行规则` / `[C] 修改一条支持原则` / `[D] 描述我自己的方向`

立即将批准的章节写入文件。

### 章节 2：氛围与气氛

**目标**：按游戏状态划分的情感目标 — 足够具体，光照艺术师可以据此工作。

对于每个主要游戏状态（例如，探索、战斗、胜利、失败、菜单 — 根据此游戏的状态调整），定义：
- 主要情感/氛围目标
- 光照特征（时间段、色温、对比度级别）
- 氛围描述词（3–5 个形容词）
- 能量级别（激烈 / 从容 / 沉思 / 等等）

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉识别声明和支柱集。询问："为此游戏的每个主要游戏状态定义氛围和气氛目标。要具体 — '黑暗和令人不安'是不够的。命名确切的情感目标、光照特征（暖/冷、高/低对比度、时间方向），以及至少一个承载氛围的视觉元素。每个游戏状态必须在视觉上与其他状态明显不同。"

立即将批准的章节写入文件。

### 章节 3：形状语言

**目标**：使此游戏世界在视觉上连贯且可辨识的几何词汇。

涵盖：
- 角色剪影哲学（缩略图大小下有多可读？每种原型的区分特征？）
- 环境几何（棱角/曲线/有机/几何 — 哪种占主导以及为什么？）
- UI 形状语法（UI 是否呼应世界美学，还是一种独特的 HUD 语言？）
- 主角形状与配角形状（什么吸引目光，什么退入背景？）

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉识别声明和氛围目标。询问："为此游戏定义形状语言。将每条形状原则与视觉识别声明和特定游戏支柱联系起来。解释这些形状选择在情感上向玩家传达了什么。"

立即将批准的章节写入文件。

### 章节 4：色彩系统

**目标**：完整的、可生产的调色板系统，同时服务于美学和传达需求。

涵盖：
- 主调色板（5–7 种颜色，带角色定义 — 不只是十六进制代码，而是每种颜色在此世界中的含义）
- 语义色彩用法（红色传达什么？金色？蓝色？白色？建立色彩词汇）
- 每个生物群落或区域的色温规则（如果游戏有不同区域）
- UI 调色板（可能与世界调色板不同 — 明确定义差异）
- 色盲安全性：哪些语义颜色需要形状/图标/声音备用方案

**Agent 委托**：通过 Task 生成 `art-director`，提供视觉识别声明和氛围目标。询问："为此游戏设计色彩系统。每个语义颜色分配都必须解释 — 为什么这种颜色在此世界中意味着危险/安全/奖励？识别哪些色彩对可能对色盲玩家造成困难，并指定需要哪些备用提示。"

立即将批准的章节写入文件。

## 阶段3：生产指南（章节 5–8）

这些章节将视觉识别转化为具体的生产规则。应该足够具体，外包团队无需额外简报即可遵循。

### 章节 5：角色设计方向

**Agent 委托**：通过 Task 生成 `art-director`，提供章节 1–4。询问："为此游戏定义角色设计方向。涵盖：玩家角色的视觉原型（如果有）、每种角色类型的区分特征规则（玩家如何一眼识别敌人/NPC/盟友？）、表情/姿势风格目标（僵硬/表现力强/写实/夸张）、以及 LOD 哲学（游戏摄像机距离下保留多少细节？）"

将批准的章节写入文件。

### 章节 6：环境设计语言

**Agent 委托**：通过 Task 生成 `art-director`，提供章节 1–4。询问："为此游戏定义环境设计语言。涵盖：建筑风格及其与世界文化/历史的关系、纹理哲学（绘制 vs. PBR vs. 风格化 — 为什么此游戏做此选择？）、道具密度规则（稀疏/密集 — 每种区域类型由什么驱动选择？）、以及环境叙事指南（哪些视觉细节应该无需文字即可讲述故事？）"

将批准的章节写入文件。

### 章节 7：UI/HUD 视觉方向

**Agent 委托**：并行生成：
- **`art-director`**：UI 的视觉风格 — 沉浸式 vs. 屏幕空间 HUD、字体方向（字体个性、粗细、大小层级）、图标风格（扁平/轮廓/插图/照片写实）、UI 元素的动画感
- **`ux-designer`**：UX 对齐检查 — 视觉方向是否支持此游戏所需的交互模式？标记艺术方向与可读性/可访问性需求之间的任何冲突。

收集两者。如果它们冲突（例如，art-director 希望精心设计的沉浸式 UI，但 ux-designer 标记它会降低战斗可读性），明确呈现冲突及双方立场。不要静默解决 — 使用 `AskUserQuestion` 让用户决定。

将批准的章节写入文件。

### 章节 8：资产标准

**Agent 委托**：并行生成：
- **`art-director`**：文件格式偏好、命名约定方向、纹理分辨率级别、LOD 级别期望、导出设置哲学
- **`technical-artist`**：引擎专属硬性约束 — 每个资产类别的面数预算、纹理内存限制、材质槽数量、导入器约束、来自 `.claude/docs/technical-preferences.md` 中性能预算的任何内容

如果任何艺术偏好与技术约束冲突（例如，art-director 希望 4K 纹理但性能预算需要移动端使用 2K），明确解决冲突 — 注明理想标准和受约束的标准，并解释权衡。资产标准中的歧义是生产成本产生的地方。

将批准的章节写入文件。

## 阶段4：参考方向（章节 9）

**目标**：精选的参考集，对每个来源具体说明借鉴什么和避免什么。

**Agent 委托**：通过 Task 生成 `art-director`，提供已完成的章节 1–8。询问："为此游戏编译参考方向。提供 3–5 个参考来源（游戏、电影、艺术风格或特定艺术家）。对于每个：命名它，精确指定要从中借鉴的视觉元素（不是'整体美学' — 是具体的技术、色彩选择或构图规则），并指定明确要避免或背离的内容（以防止'试图复制 X'的阅读）。参考应该是累加的 — 没有两个参考应该指向完全相同的方向。"

将批准的章节写入文件。

## 阶段6：收尾

在呈现下一步之前，检查项目状态：
- `design/gdd/systems-index.md` 是否存在？→ map-systems 已完成，跳过该选项
- `.claude/docs/technical-preferences.md` 是否包含已配置的引擎（非 `[TO BE CONFIGURED]`）？→ setup-engine 已完成，跳过该选项
- `design/gdd/` 是否包含任何 `*.md` 文件？→ design-system 已运行，跳过该选项
- `design/gdd/gdd-cross-review-*.md` 是否存在？→ review-all-gdds 已完成
- GDD 是否存在（见上方）？→ 包含 /consistency-check 选项

使用 `AskUserQuestion` 进行下一步。仅包含基于上方状态检查确实是下一步的选项：

**选项池 — 仅在尚未完成时包含：**
- `[_] 运行 /map-systems — 在编写 GDD 之前将概念分解为系统`（如果 systems-index.md 存在则跳过）
- `[_] 运行 /setup-engine — 配置引擎（引擎设置后可能需要重新审视资产标准）`（如果引擎已配置则跳过）
- `[_] 运行 /design-system — 开始第一个 GDD`（如果任何 GDD 存在则跳过）
- `[_] 运行 /review-all-gdds — 跨 GDD 一致性检查（技术设置关卡前必需）`（如果 gdd-cross-review-*.md 存在则跳过）
- `[_] 运行 /asset-spec — 从已批准的 GDD 生成每个资产的视觉规范和 AI 生成提示`（如果 GDD 存在则包含）
- `[_] 运行 /consistency-check — 扫描现有 GDD 与艺术圣经的视觉方向冲突`（如果 GDD 存在则包含）
- `[_] 运行 /create-architecture — 编写主架构文档（技术设置的下一步）`
- `[_] 停在这里`

仅为实际包含的选项分配字母 A、B、C…。将最有利于推进流水线的选项标记为 `（推荐）`。

> **始终包含** `/create-architecture` 和"停在这里"作为选项 — 艺术圣经完成后这些始终是有效的下一步。

## 协作协议

每个章节遵循：**问题 → 选项 → 决策 → 草案（来自 art-director Agent）→ 批准 → 写入文件**

- 永远不要在没有先生成相关 Agent 的情况下起草章节
- 批准后立即将每个章节写入文件 — 不要批量处理
- 将所有 Agent 分歧呈现给用户 — 不要静默解决 art-director 和 technical-artist 之间的冲突
- 艺术圣经是约束文档：它以视觉一致性换取对未来决策的限制。每个章节都应该感觉像是在有效地缩小解决方案空间。

## 推荐的后续步骤

艺术圣经获批后：
- 运行 `/map-systems` 在编写 GDD 之前将概念分解为游戏系统
- 如果引擎尚未配置，运行 `/setup-engine`（引擎选择后可能需要重新审视资产标准）
- 运行 `/design-system [first-system]` 开始为每个系统编写 GDD
- 一旦 GDD 存在，运行 `/consistency-check` 验证它们与艺术圣经的视觉规则
- 运行 `/create-architecture` 生成主架构文档
