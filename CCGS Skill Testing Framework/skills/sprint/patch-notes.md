# 技能测试规范：/patch-notes

## 技能摘要

`/patch-notes` 是一个 Haiku 层级的技能，从现有变更日志内容生成面向玩家的补丁说明，去除内部任务 ID 和技术术语，改用通俗语言。它只筛选对玩家相关的条目（可见功能和错误修复；内部重构被排除）。不调用任何导演关卡。该技能在持久化前询问"我可以写入 `docs/patch-notes-vX.X.md` 吗？"。裁决结果始终为 COMPLETE。

---

## 静态断言（结构）

由 `/skill-test static` 自动验证 —— 无需测试夹具。

- [ ] 具有必需的前置元数据字段：`name`、`description`、`argument-hint`、`user-invocable`、`allowed-tools`
- [ ] 具有 ≥2 个阶段标题
- [ ] 包含裁决关键词：COMPLETE
- [ ] 包含 "May I write" 语言（技能写入补丁说明文件）
- [ ] 具有下一步交接（例如，与社区经理分享）

---

## 导演关卡检查

无。补丁说明生成是一项快速的编译任务；不调用任何关卡。

## 测试用例

### 用例 1：理想路径 —— 变更日志筛选为面向玩家的条目

**测试夹具：**
- `docs/CHANGELOG.md` 存在，包含 5 个条目：
  - "Add dual-wield melee system"（功能 —— 面向玩家）
  - "Fix crash on level transition"（修复 —— 面向玩家）
  - "Add enemy patrol AI"（功能 —— 面向玩家）
  - "Refactor input handler to use event bus"（修复 —— 仅内部）
  - "Update dependency: Godot 4.6"（仅内部）
- 版本为 `v0.4.0`

**输入：** `/patch-notes v0.4.0`

**预期行为：**
1. 技能读取 `docs/CHANGELOG.md`
2. 技能筛选出 3 个面向玩家的条目；排除 2 个内部条目
3. 技能用通俗语言重写条目（无任务 ID，无技术术语）
4. 技能向用户展示草稿
5. 技能询问"我可以写入 `docs/patch-notes-v0.4.0.md` 吗？"
6. 用户批准；文件写入；裁决为 COMPLETE

**断言：**
- [ ] 补丁说明中只有 3 个条目（2 个内部条目被排除）
- [ ] 条目用通俗语言编写，不含内部任务 ID
- [ ] 文件路径匹配 `docs/patch-notes-v0.4.0.md`
- [ ] "May I write" 提示在文件写入前出现
- [ ] 写入后裁决为 COMPLETE

### 用例 2：未找到变更日志 —— 引导先运行 /changelog

**测试夹具：**
- `docs/CHANGELOG.md` 不存在

**输入：** `/patch-notes v0.4.0`

**预期行为：**
1. 技能尝试读取 `docs/CHANGELOG.md` —— 未找到
2. 技能输出："未找到变更日志 —— 先运行 /changelog 生成一个"
3. 不生成补丁说明；不写入文件

**断言：**
- [ ] 当变更日志不存在时，技能不会崩溃
- [ ] 输出明确引导用户运行 `/changelog`
- [ ] 不出现 "May I write" 提示（无内容可写入）
- [ ] 裁决为 BLOCKED（依赖未满足）

**测试夹具：**

2. 技能检测到 `design/community/tone-guide.md` 的语气指导

- [ ] 技能检查 `design/` 中的社区或语气指导文件
- [ ] 语气指导内容影响补丁说明条目的措辞
- [ ] 输出在适用情况下反映主动语态和积极语气
- [ ] 技能记录已应用语气指导

## 协议合规性

- [ ] 在生成补丁说明前读取 `docs/CHANGELOG.md`
- [ ] 将条目筛选为仅面向玩家的项目
- [ ] 用通俗语言重写条目，不含内部 ID
- [ ] 在写入补丁说明文件前始终询问 "May I write"
- [ ] 不调用导演关卡
- [ ] 在 Haiku 模型层级上运行（快速、低成本）

## 覆盖说明

- 未测试所有变更日志条目都是内部条目（零个面向玩家项目）的情况；行为是一个空的补丁说明草稿并附带警告。
- 从变更日志标题解析版本号是此处未验证的实现细节。
- 用例 5 中记录的社区经理咨询是建议性的；单独的 Skill 或手动审查处理该步骤。
