# 项目阶段检测

此 skill 扫描您的项目以确定其当前开发阶段、工件的完整性和需要注意的差距。它在以下情况特别有用：
- 开始使用现有项目
- 加入代码库
- 检查里程碑之前缺少什么
- 了解"我们在哪里？"

---

## 工作流程

### 1. 扫描关键目录

分析项目结构和内容：

**设计文档** (`design/`)：
- 统计 `design/gdd/*.md` 中的 GDD 文件数量
- 检查 game-concept.md、game-pillars.md、systems-index.md 是否存在
- 如果 systems-index.md 存在，统计总系统数 vs 已设计系统数
- 分析完整性（概述、详细设计、边界情况等）
- 统计 `design/narrative/` 中的叙事文档数量
- 统计 `design/levels/` 中的关卡设计数量

**源代码** (`src/`)：
- 统计源文件数量（与语言无关）
- 识别主要系统（有 5+ 文件的目录）
- 检查是否存在 core/、gameplay/、ai/、networking/、ui/ 目录
- 估计代码行数（大致规模）

**生产工件** (`production/`)：
- 检查是否存在活动冲刺计划
- 查找里程碑定义
- 查找路线图文档

**原型** (`prototypes/`)：
- 统计原型目录数量
- 检查 README（有文档记录 vs 无文档记录）
- 评估原型是已归档还是活跃的

**架构文档** (`docs/architecture/`)：
- 统计 ADR（架构决策记录）数量
- 检查是否存在概述/索引文档

**测试** (`tests/`)：
- 统计测试文件数量
- 估计测试覆盖率（粗略启发式）

### 2. 分类项目阶段

基于扫描的工件确定阶段。首先检查 `production/stage.txt` — 
如果存在，使用其值（来自 `/gate-check` 的显式覆盖）。否则，
使用这些启发式方法自动检测（从最先进的向后检查）：

| 阶段 | 指标 |
|-------|-----------|
| **概念** | 无游戏概念文档，头脑风暴阶段 |
| **系统设计** | 游戏概念存在，系统索引缺失或不完整 |
| **技术设置** | 系统索引存在，引擎未配置 |
| **预生产** | 引擎已配置，`src/` 有 <10 个源文件 |
| **生产** | `src/` 有 10+ 个源文件，活跃开发中 |
| **润色** | 仅限显式（由 `/gate-check` 生产 → 润色门设置） |
| **发布** | 仅限显式（由 `/gate-check` 润色 → 发布门设置） |

### 3. 协作差距识别

**不要**只列出缺失的文件。相反，**提出澄清问题**：

- "我看到战斗代码 (`src/gameplay/combat/`) 但没有 `design/gdd/combat-system.md`。这是先进行原型制作的，还是应该反向文档化？"
- "您有 15 个 ADR 但没有架构概述。我应该创建一个来帮助新贡献者吗？"
- "`production/` 中没有冲刺计划。您在其他地方（Jira、Trello 等）跟踪工作吗？"
- "我发现了一个游戏概念但没有系统索引。您已经将概念分解为单个系统，还是应该运行 `/map-systems`？"
- "原型目录有 3 个没有 README 的项目。这些是实验，还是需要文档？"

### 4. 生成阶段报告

使用模板：`.claude/docs/templates/project-stage-report.md`

**报告结构**：
```markdown
# 项目阶段分析

**日期**：[日期]
**阶段**：[概念/系统设计/技术设置/预生产/生产/润色/发布]
**阶段置信度**：[通过 — 明确检测 / 顾虑 — 信号模糊 / 失败 — 关键差距阻碍进展]

## 完整性概述
- 设计：[X%]（[N] 个文档，[差距]）
- 代码：[X%]（[N] 个文件，[系统]）
- 架构：[X%]（[N] 个 ADR，[差距]）
- 生产：[X%]（[状态]）
- 测试：[X%]（[覆盖率估计]）

## 识别的差距
1. [差距描述 + 澄清问题]
2. [差距描述 + 澄清问题]

## 推荐的后续步骤
[基于阶段和角色的优先级排序列表]
```

### 5. 角色过滤推荐（可选）

如果用户提供了角色参数（例如，`/project-stage-detect programmer`）：

**程序员**：
- 关注架构文档、测试覆盖率、缺失的 ADR
- 代码与文档的差距

**设计师**：
- 关注 GDD 完整性、缺失的设计章节
- 原型文档

**制作人**：
- 关注冲刺计划、里程碑跟踪、路线图
- 跨团队协调文档

**通用**（无角色）：
- 所有差距的整体视图
- 跨领域的最高优先级项目

### 6. 写入前请求批准

**协作协议**：
```
我已经分析了您的项目。以下是我发现的：

[显示摘要]

识别的差距：
1. [差距 1 + 问题]
2. [差距 2 + 问题]

推荐的后续步骤：
- [优先级 1]
- [优先级 2]
- [优先级 3]

我可以将完整的阶段分析写入 production/project-stage-report.md 吗？
```

等待用户批准后再创建文件。

## 后续行动

生成报告后，建议相关的后续步骤：

- **概念存在但没有系统索引？** → `/map-systems` 分解为系统
- **缺少设计文档？** → `/reverse-document design src/[system]`
- **缺少架构文档？** → `/architecture-decision` 或 `/reverse-document architecture`
- **原型需要文档？** → `/reverse-document concept prototypes/[name]`
- **无冲刺计划？** → `/sprint-plan`
- **接近里程碑？** → `/milestone-review`

---

## 协作协议

此 skill 遵循协作设计原则：

1. **先提问**：对差距提问，不要假设
2. **展示选项**："我应该创建 X，还是它在其他地方跟踪？"
3. **用户决定**：等待指示
4. **展示草案**：显示报告摘要
5. **获取批准**："我可以写入 production/project-stage-report.md 吗？"

**绝不要**静默写入文件。**始终**展示发现并在创建工件之前询问。
